# Design & Implementation Review

codex に **設計レビュー → 実装 → 実装レビュー** の 2 フェーズを通させるワークフロー。

「codex に設計レビューと実装レビューをやってもらって」という定型依頼をスキル化したもの。コードを書く前に設計方針を codex に通し、合意できてから実装に入り、実装後は差分を codex に通す。

## 引数

`$ARGUMENTS` は自由テキスト。実装する機能/修正の説明、対象ファイル、重点的に見てほしい観点を含められる。

```
/design-impl-review SyncEngine の競合解決ロジックを再設計したい
/design-impl-review issues/042-sync-conflict.md の方針で実装する
/design-impl-review                → 直前の会話で固まっている設計方針を対象にする
```

## 全体の流れ

```
[Phase 1] 設計レビュー   … 実装前に設計方針/計画を codex に通す（合意できるまでループ）
   ↓ 合意
[Phase 2] 実装          … 合意した設計どおりに実装する
   ↓
[Phase 3] 実装レビュー   … 差分を codex-review スキルに通す（指摘がなくなるまでループ）
```

**Phase 1 を飛ばして実装に入らないこと。** 設計レビューで合意を取ってから実装するのがこのスキルの本質。

---

## Phase 1: 設計レビュー（実装前）

### 1-1. 設計方針をまとめる

実装に入る前に、これから書くコードの設計方針・アプローチ・変更計画を言語化する。以下を含める:

- **目的**: 何を解決するか（不具合 / 機能追加 / リファクタ）
- **不変条件（Invariant）**: 壊してはいけない契約（例: deep link は失われない、同一ファイルの同一性は一意）
- **アプローチ**: どのレイヤーに何を追加/変更するか、責務の分担
- **失敗モード**: 順序競合・再送・二重実行・部分失敗・再起動をどう吸収するか
- **影響範囲**: 触るファイル / 既存コードとの整合
- **代替案を捨てた理由**: なぜこの設計を選んだか（あれば）

### 1-2. 設計方針を ./tmp に書き出す

```bash
mkdir -p ./tmp
stamp="$(date +%Y%m%d-%H%M%S).$$"
plan_out="./tmp/design-plan.$stamp.md"
review_out="./tmp/design-review.$stamp.md"
```

`$plan_out` に 1-1 でまとめた設計方針を書く。`$ARGUMENTS` に issue ファイルや既存の計画があればそれを使う。

- `/tmp` ではなく必ず `./tmp` を使う
- 出力ファイルは毎回ユニークなパスにする

### 1-3. codex に設計レビューを依頼する

差分がまだないため、selector ではなく `codex exec -s read-only` を使い、リポジトリの現状コードと照らして設計をレビューさせる。

```bash
command codex exec -s read-only --full-auto --ephemeral -o "$review_out" \
  "$(cat <<'EOF'
以下の設計方針/変更計画を、実装に入る前にレビューして。リポジトリの現状コードを読み、現状と照らして設計上の問題を指摘する。

## 重点観点
- 責務分離・依存方向・レイヤー配置が妥当か（God クラス化の予兆を含む）
- 不変条件が崩れる経路はないか
- 失敗モード（順序競合・再送・二重実行・部分失敗・再起動）の取りこぼし
- 既存実装との不整合・重複（同じロジックが別の場所に既にある等）
- 過剰設計 / 過少設計、場当たり的な条件分岐への依存
- この方針で本当に根本原因が潰れるか（症状への対処になっていないか）

## 出力形式
問題があるものだけを重要度順（P1=必須 / P2=推奨 / P3=提案）に列挙し、各項目に対象（file:line または計画の該当箇所）、理由、代替案を書く。要約や称賛は不要。

---

EOF
)
$(cat "$plan_out")"
```

- `command` プレフィックス必須（zsh の関数オーバーライドを回避）
- 常に `--full-auto --ephemeral -o "$review_out"` を付与する
- コマンド実行ツールのタイムアウトは 900000ms（15分）に設定する
- `$ARGUMENTS` に重点観点があれば「重点観点」セクションの末尾に追記する

### 1-4. 結果の読み取りと反映

1. まず `"$review_out"` を読む。空なら codex コマンドの stdout / stderr を確認する
2. 指摘を以下に分けて報告する:
   - **設計に取り込むべきもの**（P1/P2 = 不変条件破り、失敗モード取りこぼし、既存実装との不整合など）
   - **任意・後回し**（P3）
   - **指摘なし** ならその旨
3. P1/P2 の指摘は設計方針に反映する（`$plan_out` を更新する）
4. 設計を大きく変えた場合は、更新した方針で 1-3 を再実行する（指摘が落ち着くまでループ）

### 1-5. 合意を取る

更新後の設計方針をユーザーに提示し、**実装に進んでよいか確認する**。ユーザーの承認なしに Phase 2 へ進まない。

### 1-6. 一時ファイルの削除

設計レビューを報告し、合意が取れた直後に、このフェーズで書き出した `$plan_out` / `$review_out` を削除する。

```bash
rm -f "$plan_out" "$review_out"
```

- 削除対象は今回作った 2 ファイルのみ。`./tmp/*.md` を一括削除しない
- ユーザーが「残しといて」と明示した場合のみスキップ

---

## Phase 2: 実装

合意した設計方針どおりに実装する。

- Phase 1 で codex から指摘された点は実装でも守る
- 実装中に設計の前提が崩れたら（合意した方針では解けないと分かったら）、勝手に方針を変えず Phase 1 に戻ってユーザーに再合意を取る
- `~/.claude/CLAUDE.md` の共通ルール（God クラス回避、重複コードの抽出、構造的修正優先など）に従う

---

## Phase 3: 実装レビュー（実装後）

実装した差分を codex に通す。

### 3-1. codex-review スキルを実行

`~/.claude/skills/codex-review/SKILL.md` を Read し、その手順に従って実装差分をレビューする。

- 未コミット変更なら `codex exec review --uncommitted` がデフォルト
- 重点観点があれば codex-review の fallback モードでプロンプトに追記する
- 出力はまず `./tmp` の `-o` ファイルを読み、空なら codex コマンドの stdout / stderr を使う

### 3-2. 指摘への対応

- **P1/P2**: コードを修正する。場当たり的なパッチではなく構造的に直す
- **P3**: 取り込むか、issue 化するか、認識のみかを判断する
- 修正したら必要に応じて再レビュー（指摘が落ち着くまでループ）

---

## ルール

- **Phase 1（設計レビュー）を必ず先に通す。** 実装前に codex の設計レビューで合意を取ってから Phase 2 に進む
- 設計レビューと実装レビューはどちらも codex を使い、コードを破壊的に変更しない（レビューは read-only）
- `command codex` を使うこと（`codex` 直接呼び出しは zsh 関数オーバーライドでエラーになる場合がある）
- 設計レビューは差分がないため `codex exec -s read-only` を使う（selector は使わない）
- 常に `--full-auto --ephemeral -o "$review_out"` を付与する
- まず `-o` の出力ファイルを読み、空なら stdout / stderr を fallback として使う
- レビュー結果はそのままユーザーに見せる（要約しすぎない）
- `/tmp` は使わず、出力ファイルは必ず `./tmp` に置く。報告後、このフェーズで書き出したファイルだけを `rm -f` で削除する
- コマンド実行ツールのタイムアウトは 900000ms（15分）に設定する
- **構造的修正優先**: 設計レビュー・実装レビューいずれの指摘でも、修正方針は中長期的に改修を続けることを前提とした構造的な方針を優先する。場当たり的な条件分岐やワークアラウンドを修正案として提示しない
- 実装レビューの詳細手順は `codex-review` スキルに委ねる（このスキルでは設計レビューと全体フローの管理に集中する）

## 関連スキル

- `~/.claude/skills/codex-review/SKILL.md` — Phase 3 で使う実装（コード変更）レビュー
- `~/.claude/skills/cross-review/SKILL.md` — 有識者エージェント + codex の複数視点レビュー（実装レビューをより厚くしたい場合）
- `~/.claude/skills/review-loop/SKILL.md` — GitHub PR 上で codex と往復レビューするループ
