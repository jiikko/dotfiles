# Codex Review

`codex review` コマンドを使って、コード変更に対する Codex のレビューを取得する。

## 引数

`$ARGUMENTS` は自由テキスト。commit ID や重点的に見てほしい観点を含められる。

```
/codex-review                                    → 未コミットの変更をレビュー
/codex-review abc1234                            → 特定コミットをレビュー
/codex-review abc1234 削除ロジックの安全性を重点的に見て
/codex-review 同期周りのモデリングが適切か確認して
```

## 手順

### 1. 引数のパース

`$ARGUMENTS` から以下を抽出する:

- **commit ID**: 7〜40文字の hex 文字列（`[0-9a-f]{7,40}`）があれば `--commit {sha}` として使用
- **カスタム指示**: commit ID 以外のテキストがあれば、レビュープロンプトの末尾に追加
- **両方なし**: `--uncommitted` で未コミット変更をレビュー

### 2. レビューの実行

```bash
codex review [--uncommitted | --commit {sha}] "$(cat <<'PROMPT'
あなたは経験豊富なコードレビュアーとして、提出されたコード変更をレビューしてください。

## レビュー観点（優先度順）
1. **設計・モデリング**: この変更は長期運用を見据えた適切なモデリングになっているか。ドメインの概念が正しくモデル化されているか。責務の境界は適切か。将来の変更容易性を損なう構造的な問題はないか。安易な条件分岐の追加で済ませていないか（構造で解決すべきでないか）
2. **正当性**: バグ、ロジックエラー、エッジケースの見落とし
3. **セキュリティ**: インジェクション、認証/認可の欠陥、機密情報の露出
4. **パフォーマンス**: N+1、不要な計算、メモリリーク
5. **保守性**: 過度な結合、命名の不明瞭さ、不要な抽象化

## 出力形式
- 各指摘にはファイルパスと行番号を明記すること
- 重大度を P1（必須修正）/ P2（推奨修正）/ P3（提案）で分類すること
- スタイルの好みだけの指摘はしないこと
- 最後に全体の正当性判定（correct / incorrect）と確信度（0-1）を提示すること

{カスタム指示があればここに追加}

)"
```

### 3. 結果の報告

Codex の出力をユーザーに報告する。指摘がある場合は重大度順に整理する。

## プロジェクト固有のコンテキスト

CLAUDE.md が存在する場合、そのプロジェクトのルール（禁止パターン、設計原則など）をプロンプトに追記して、プロジェクト固有の観点でもレビューさせる。

例: Swift プロジェクトなら以下を追記:
```
## プロジェクト固有ルール
- @StateObject 禁止 → @Observable + @State を使う
- try? によるエラー握りつぶし禁止
- DispatchQueue.main.async 禁止 → Task { @MainActor in } を使う
```

## ルール

- レビュー結果はそのままユーザーに見せる（要約しすぎない）
- P1/P2 の指摘があれば、修正するか確認する
- `codex review` はリードオンリーなので、コードを変更しない
- タイムアウトは 300秒（5分）に設定する
