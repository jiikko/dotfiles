# 1 回目の修正仮説が外れたら、2 発目を打つ前に「観測を増やす」を選ぶ

## ルール

- **自分の仮説に基づく 1 回目の修正が「効かなかった」と分かった瞬間、2 つ目の修正を**
  **コードに打つ前に、まず観測手段 (ログ / 計測 / wire capture / A-B プローブ) を仕込んで**
  **「見えていない事実」を可視化する** ことを選ぶ
- **情報を増やさないまま 2 発目の blind fix を打つことを禁止する**。「次はこれで直るかも」で
  別の修正を試すのは、確度の低い推測の連鎖であり、当たっても理由が分からず、外れればコードが汚れる
- 「1 回外れた」= **その不具合の前提 (実際の振る舞い) が自分には見えていない** という証拠。
  次にやるべきは「別の修正案を考える」ではなく「**何が見えていないかを観測する**」

## なぜこのルールが必要か (obaket Dropbox -1005、2026-06-09 の直接の起源)

Dropbox の画像 download が `URLError.networkConnectionLost` (-1005) で失敗する bug で、私は
**app 層ログ (`-1005` という結果だけ) を頼りに推測 fix を 2 連続で外した**:

1. 仮説 A「HTTP/2 connection-reuse race + CFNetwork は POST を透過 retry しない」→ transport
   retry を実装 → **実機で 3 retry 全滅**。connection-reuse 説は否定。revert。
2. 仮説 B「body-method POST を `httpBody` unset で task-delegate streaming 送信」→ `httpBody =
   Data()` を実装 → **実機で再発**。body framing 説も否定。revert。

両方とも **wire レベル (URLSession が実際に何を送り、サーバがどの時点でなぜ接続を切ったか) が**
**見えないまま** app 層ログだけで推測していた。ユーザーに「治っていない / 状況把握できてるのか /
codex に引き継ぎたい」と 2 度言わせ、commit 4 本 (fix 2 + revert 2) を無駄に積んだ。

**1 回目 (retry) が外れた時点で**、別の修正を考えるのではなく `CFNETWORK_DIAGNOSTICS=3` /
`URLSessionTaskMetrics` のログ / buffered 経路との A-B を仕込んでいれば、2 発目の無駄打ちと
revert を避け、観測データから真因に最短で寄れた。私の損失は「推測 fix → 実機検証待ち → 外れ →
revert」のサイクルを 2 周したこと。観測を仕込むコスト (数十分) の方が圧倒的に安い。

## 「不具合対応の原則」との関係 (矛盾しない、むしろ操作化)

`~/.claude/CLAUDE.md`「不具合対応の原則」は **「追加ログは最後の手段。まず不変条件を言語化し**
**設計で潰す」** と言っている。本ルールはこれと矛盾しない。違いは **タイミングと目的**:

| | 「不具合対応の原則」が戒めること | 本ルールが推奨すること |
|---|---|---|
| タイミング | 調査の **最初** に「ログを足して現象を追う」 | 構造的仮説の **1 回目が外れた後** |
| 目的 | 症状を場当たりに追う print の乱射 | **見えていない事実 (状態遷移 / 境界 I/O / wire) の観測** |
| 何を見るか | 現象の断片 | 不変条件が破れている箇所 / 失敗フェーズ |

= 「最初から print を撒くな」と「1 回外したら観測を増やせ」は両立する。後者は前者の例外ではなく、
「不具合対応の原則」自身の **「必要なら『イベント／状態遷移』が観測できる設計にする」** を
**いつ発動するか (= 1 回目の仮説が外れた瞬間)** に落とし込んだもの。観測の質も原則どおり
「症状を追う print」ではなく **不変条件・状態遷移・境界の入出力・wire** を可視化する設計にする。

## 採用パターン

### 1 回目の仮説 fix が外れたら踏むステップ

```
1. STOP。2 つ目の修正案を考え始めない。
2. 問いを変える: 「次に何を直すか」ではなく「何が見えていないか」。
   - 失敗は結果しか見えていないか? (例: -1005 という code だけ)
   - 成功している隣接経路との差分の、どこが観測できていないか?
   - 前提 (「X は動く」) は今この環境で検証済みか? 古いログ由来の思い込みでないか?
3. 観測手段を仕込む (下表から選ぶ)。
4. 観測データで仮説を作り直す。データが仮説と矛盾したらデータを優先する。
5. 観測で前提が確定してから、はじめて 2 つ目の修正を打つ。
```

### 観測手段の引き出し (症状でなく構造を見るもの)

| 観測したいもの | 手段 |
|---|---|
| 状態遷移 / イベント順序 | 構造化ログ (state machine の遷移を 1 行ずつ。`print` でなく `NSLog`/`os_log`、`macos-debug-logging` 規約) |
| 境界の入出力 | request/response・関数の引数/戻り値・契約違反を境界でログ |
| ネットワーク wire | `CFNETWORK_DIAGNOSTICS=3` / `URLSessionTaskMetrics` / proxy (Proxyman 等) で byte 比較 |
| 成功 vs 失敗の差分 | 動いている隣接経路と **同条件で A-B プローブ** (例: buffered vs streaming を同 file で投げ比べ) |
| 前提の再検証 | 「X は動く」を **今この環境で実測** し直す (過去ログ・別条件の思い込みを排除) |
| 再現条件 | 毎回か / 初回からか / 特定入力か を切り分けるログ |

### 観測を入れた後の判断

- 観測データが **真因を指したら** → そのデータに基づく 1 発で直す (推測でなく観測駆動)
- 観測しても **まだ分からない / framework 内部挙動** → `escalate-to-forge-after-failed-tries.md`
  で forge / 別視点に escalate (観測データを手土産にする = 専門家の収束が速い)
- 観測手段自体を仕込むのが重い (実機専用 / 再現困難) → ユーザーに「この観測を取ってほしい」と
  具体手順を渡す (= blind fix を投げ続けるより誠実)

## escalate-to-forge との順序関係

```
1 回目の仮説 fix が外れた
   │
   ├─→ 【本ルール】まず観測を増やす (見えていない事実を可視化)
   │       │
   │       ├─ 観測で真因判明 → 観測駆動で 1 発修正
   │       └─ 観測しても不明 / framework 内部 → ↓
   │
   └─→ 【escalate-to-forge】別視点 (forge Maximum/Ultra) に escalate
           (観測データを添えると専門家の収束が速い)
```

両ルールは「**自分の推測の連鎖を断つ**」思想で一貫する。本ルールは「観測を増やす」、forge は
「別の人格・視点を入れる」。**順序は観測が先**: 観測なしで forge に投げると、専門家も同じ
app 層ログから推測することになり (実際 obaket では forge の 3 専門家も wire が見えず「device
確認が前提」で止まった)、観測データがあれば収束が段違いに速い。

## 例外 (本ルールを発動しなくてよい)

- typo / build error / lint のような **結果が自明** なミス (観測不要、直せばわかる)
- 自分が以前直したパターンの **完全な再発** (観測済みの既知問題)
- ユーザーが「この 1 点だけ確認して」と **観測でなく特定アクションを明示** している
- 1 回目がそもそも **検証 (= 仮説の確認) であって修正でない** 場合 (それは観測の一種)

## やること / やらないこと

- ✓ 1 回目の仮説 fix が「効かない」と分かったら、2 発目の前に観測を仕込む
- ✓ 観測は「状態遷移 / 境界 I/O / wire / A-B 差分」など **構造**を見るものにする
- ✓ 「X は動く」という前提を、過去ログでなく **今この環境で実測**し直す
- ✓ 観測しても不明なら forge へ escalate (観測データを手土産に)
- ✓ 実機専用で自分が観測できないなら、ユーザーに具体的な観測手順を渡す
- ✗ 情報を増やさずに 2 発目・3 発目の blind fix を打つ (推測の連鎖)
- ✗ 「次はこれで直るかも」で別の修正案に飛びつく (1 回外した = 前提が見えていない証拠)
- ✗ 古いログ・別条件の結果を「現環境でも真」と思い込む
- ✗ 効かなかった修正を revert せず残す (`不具合対応の原則` と重複だが再掲)
- ✗ 症状を場当たりに追う `print` の乱射 (観測は構造を見る設計にする)
