# 1 回目の修正仮説が外れたら、2 発目を打つ前に「観測を増やす」を選ぶ

## ルール

- **1 回目の仮説 fix が「効かなかった」と分かったら、2 つ目の修正をコードに打つ前に、観測手段 (ログ / 計測 / wire capture / A-B プローブ) を仕込んで「見えていない事実」を可視化する**
- **情報を増やさないまま 2 発目の blind fix を打つことを禁止する**。「次はこれで直るかも」は確度の低い推測の連鎖であり、当たっても理由が分からず、外れればコードが汚れる
- 「1 回外れた」= その不具合の実際の振る舞いが自分には見えていない証拠。次にやるべきは「別の修正案」ではなく「**何が見えていないかの観測**」

## なぜ (起源: obaket issue 227 Dropbox -1005, 2026-06-09)

`URLError -1005` の bug で、wire レベル (URLSession が実際に何を送り、サーバがなぜ接続を切ったか) が見えないまま app 層ログだけで推測 fix を 2 連続で外し、commit 4 本 (fix 2 + revert 2) と実機検証 2 周を無駄にした。1 回目が外れた時点で `CFNETWORK_DIAGNOSTICS=3` / `URLSessionTaskMetrics` / 成功経路との A-B プローブを仕込んでいれば最短で真因に寄れた。観測を仕込むコスト (数十分) の方が圧倒的に安い。

## 「不具合対応の原則」(CLAUDE.md) との関係

矛盾しない。原則が戒めるのは「調査の**最初**に print を撒いて症状を追う」こと。本ルールは「構造的仮説の 1 回目が**外れた後**」に発動し、症状でなく **不変条件・状態遷移・境界 I/O・wire という構造** を観測する。原則の「必要なら『イベント／状態遷移』が観測できる設計にする」の発動タイミングを定めたもの。

## 手順 (1 回目の仮説 fix が外れたら)

1. STOP。2 つ目の修正案を考え始めない
2. 問いを変える: 「次に何を直すか」→「何が見えていないか」。失敗は結果しか見えていないか? 成功している隣接経路との差分のどこが観測できていないか? 「X は動く」という前提は今この環境で検証済みか?
3. 観測手段を仕込む (下表)
4. 観測データで仮説を作り直す。データが仮説と矛盾したらデータを優先する
5. 前提が確定してから、はじめて 2 つ目の修正を打つ

| 観測したいもの | 手段 |
|---|---|
| 状態遷移 / イベント順序 | 構造化ログ (`print` でなく `NSLog`/`os_log`。tee pipe で block-buffered になり出ないため) |
| 境界の入出力 | request/response・引数/戻り値・契約違反を境界でログ |
| ネットワーク wire | `CFNETWORK_DIAGNOSTICS=3` / `URLSessionTaskMetrics` / proxy で byte 比較 |
| 成功 vs 失敗の差分 | 動いている隣接経路と同条件で A-B プローブ |
| 前提の再検証 | 「X は動く」を今この環境で実測し直す (過去ログの思い込みを排除) |
| 再現条件 | 毎回か / 初回からか / 特定入力か を切り分ける |

## 観測後の判断

- 真因を指したら → 観測駆動で 1 発修正
- まだ不明 / framework 内部挙動 → [`escalate-to-forge-after-failed-tries.md`](escalate-to-forge-after-failed-tries.md) へ。観測データを手土産にすると専門家の収束が段違いに速い (観測なしで投げると専門家も同じ app 層ログから推測することになる)
- 観測自体を仕込むのが重い (実機専用 / 再現困難) → ユーザーに具体的な観測手順を渡す (blind fix を投げ続けるより誠実)

## 例外 (発動不要)

- typo / build error / lint のような結果が自明なミス
- 以前直したパターンの完全な再発 (観測済みの既知問題)
- ユーザーが特定アクションを明示している場合
- 1 回目がそもそも検証 (仮説の確認) であって修正でない場合

## やること / やらないこと

- ✓ 1 回目が外れたら 2 発目の前に観測を仕込む (構造を見る設計で)
- ✓ 前提を今この環境で実測し直す
- ✓ 効かなかった修正は revert する（CLAUDE.md「不具合対応の原則」の一項。ここでは再掲であり一次情報は CLAUDE.md）
- ✗ 情報を増やさず 2 発目・3 発目の blind fix を打つ
- ✗ 古いログ・別条件の結果を「現環境でも真」と思い込む
- ✗ 症状を場当たりに追う print の乱射
