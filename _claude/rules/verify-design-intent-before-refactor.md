# リファクタ / 複雑度改善を提案する前に「既に意図的に選ばれた設計」でないか確認する

> **これは常時回すチェックではなく、トリガー型ルール。**
> 「このコード複雑だな / 消したい / 分解したい / 統合できそう / proxy 多すぎ」と
> *第一印象で構造を変えたくなった瞬間* にだけ発動する。何も感じていない時に下記の
> 5 確認を回す必要はない。逆に「複雑そう」と感じたら、手を動かす前に必ず発動させる。

## ルール

- **「このコードは複雑すぎる / 分解すべき / proxy が多い / facade が肥大化している」と判断する前に、その構造が既に意図的に選ばれたものでないかを確認する**。確認すべきは 5 つ:
  1. **ドキュメント (CLAUDE.md / README / docstring)** に「意図的にこうした」と明記されていないか
  2. **既存 issue / PR / 設計メモ (完了済み含む)** で同じ分解判断が既になされ、「やらない」と決定されていないか
  3. **実 callsite** を grep して、提案が breaking change になるか / 既存の呼び出し規約を崩さないか
  4. **可視性 (private / internal 等のアクセス制御)** を確認して、「API を変えずに論理分割」が本当に可能か
  5. **行数ではなく認知負荷** で測る。大ファイルが domain 複雑度 (仕様・プロトコル・状態機械の写像) 由来なら、分解は downgrade になりうる
- **speculative refactor (先回り分解) より「実変更 trigger 待ち」を default にする**。「将来困りそうだから今分解」ではなく「実際にそのコードを触る変更が来た時に境界を再評価」
- 改善の **効能を誇張しない**。「構造的に防止できる」と書く前に、本当に全経路を防げるか確認する (= 一部しか防げないなら「探索範囲を縮小する」と正確に書く)

## なぜこのルールが必要か

「複雑度が高い箇所を列挙して改善案を出す」タスクで、私 (claudecode) は **確認を怠った先回り判断** を量産しやすい。典型的な空振りパターン:

| 空振りパターン | 実際に起きていたこと |
|---|---|
| 「proxy / forward が多い = 認知負荷 = 削除すべき」と短絡 | ドキュメントに「proxy 経由は callsite を一度に書き換えないための意図的設計」と明記済だった (読み落とし) |
| 「この責務群を全部別クラスに抽出すべき」と提案 | 過去の issue で「行数削減だけの一斉分割はしない、実需要で判断する」と既に決定済だった (判断逆転) |
| 「API を変えずに extension で論理分割できる」と主張 | 対象が private member に依存しており、別ファイルに移すと可視性を緩める必要があった (cosmetic split にならない) |
| 大ファイル (行数) を「複雑」と評価 | 中身は仕様 (署名アルゴリズム / プロトコル定義 / 状態機械) の写像で、分解すると逆に読みにくくなる domain 複雑度だった |
| 「この分解で追従漏れを構造的に防止できる」と書く | 実際には一部の経路しか集約できず、残りは手動更新が残る (効能の誇張) |

これらは **コードを読む前に / 既存判断を確認する前に「複雑そう」という第一印象で提案** すると必ず起きる。外部レビュー (codex 等) を 1 回通すだけで、こうした空振りはまとめて検出できることが多い。確認のコストは小さく、確認なしで実装に進んだ時の手戻り (意図的設計を壊す・既存判断を逆転する・可視性で詰まる・domain 複雑度を downgrade させる) は大きい。

**このルールの効果は「やらなかったこと」に宿るため見えにくく、形骸化しやすい。** 先回り分解を *しなかった* 価値は計測できず、守っても誰も褒めない。破って意図的設計を壊した時だけ顕在化する。だからこそ「気づき任せ」にせず、トリガー型の明示ルールとして固定する。「複雑そう」と感じた瞬間が、このルールを思い出す唯一のフックである。

## 採用パターン

### リファクタ / 複雑度改善を考えた瞬間のチェックリスト

```
「ここ複雑だな、分解したいな」と思ったら、提案・実装の前に:

1. [ ] そのディレクトリ / ファイルのドキュメント (CLAUDE.md 等) を読む
       → 「意図的に proxy / facade / 同居にしている」と書いてないか?
2. [ ] docstring / 直近コメントを読む
       → 「issue NNN でこう決めた」「callsite migration を避けるため」等の理由がないか?
3. [ ] 関連しそうな issue / PR を検索 (open + 完了済み 両方)
       → 同じ分解が既に検討され「やらない」と決定されていないか?
4. [ ] 提案が触る symbol の callsite を grep
       → breaking change か? 既存の呼び出し規約を壊さないか?
5. [ ] 移動・分割したい member の可視性を確認
       → private member を別 file/type に移すと可視性変更が要らないか?
6. [ ] 行数ではなく「読む時に何回 jump するか」「追加時に何箇所触るか」で測る
       → 大ファイルが仕様の写像なら分解は逆効果
```

1 つでも「意図的にこうしている」「既に判断済」が見つかったら、**提案を取り下げるか、その制約を踏まえて scope を絞る**。

### speculative refactor を避け、trigger 待ちにする

- ✗ 「将来この型が増えたら困るから、今のうちにファイル分割しておく」
- ✓ 「次に実際にこの構造を触る変更 (機能追加 / 新ケース追加) が来た時に、サイズと集中度を再評価する」と trigger を明記して watch に置く

trigger の例: 「次にこの facade を触る機能追加」「N 個目の sub-component を足す時」「分岐対象が 2 つ以上になった時」。**実需要が来るまで構造を凍結**する。

### 効能を正確に書く

- ✗ 「この分解で追従漏れを構造的に防止できる」(= 一部しか防げないのに全防止と誇張)
- ✓ 「この分解で漏れの探索範囲を局所化できる。ただし別の owner が持つ state は引き続き手動更新が必要」

## このルールと「自律改善」ルールの関係

`~/.claude/CLAUDE.md` の「コード変更時の自律改善」は「重複コード / 死にコード / 責務混在を見つけたら **変更と同時に直す**」と指示している。本ルールはその **逆方向の制約**:

- 自律改善が対象にするのは「**触っているコードの中で**明らかに劣化している部分」(= 重複・到達不能・混在)
- 本ルールが止めるのは「**依頼と無関係に**『複雑そう』と感じて先回りで分解する」speculative refactor

= 「触ったコードを前より良くする」は続ける。「触ってもいないコードを複雑そうだから分解する」は確認なしにやらない。**「改善できそう」と「改善すべき」は違う**。

### 境界が紛らわしい時の分かれ目

実運用ではこの 2 ルールの境界が曖昧になる。判断の目安:

| 状況 | どちら | 理由 |
|---|---|---|
| 変更した関数内に到達不能コードがある | 自律改善 (消す) | 触っている範囲・劣化が明確 |
| 自分の変更で使われなくなった分岐 / mock / 定数が残った | 自律改善 (消す) | 自分の変更が生んだ死にコード |
| 別ファイル・別レイヤーに似たコードがある | verify-design-intent (触らない) | 依頼外。別目的の可能性。trigger 待ち |
| 「ついでにこの大きいファイルも割っておきたい」 | verify-design-intent (止まる) | 触ってもいない・第一印象 |

迷ったら **「今の依頼で *必然的に* 触る範囲か?」で切る**。必然なら改善し、「ついで」なら凍結して trigger 待ちに回す。

## やること / やらないこと

- ✓ refactor 提案の前にドキュメント / docstring / 既存 issue / callsite / 可視性を確認
- ✓ speculative refactor は「実変更 trigger 待ち」の watch にして trigger を明記
- ✓ 行数ではなく jump 数 / 追加時の touch 箇所数で認知負荷を測る
- ✓ 改善の効能は「全防止」ではなく実際に防げる範囲を正確に書く
- ✓ 複雑度評価 / refactor 提案は外部レビュー (codex 等) を通す ([[issue-creation-codex-review]] と併用)
- ✗ ドキュメントに「意図的」と書いてある構造を「複雑」と判定して分解提案
- ✗ 既存 issue で「やらない」と決定済の分解を逆転提案
- ✗ 「API 変えず論理分割」を可視性確認なしで主張
- ✗ domain 複雑度 (仕様の写像) を行数で「複雑」と誤判定して分解
- ✗ 改善の効能を誇張する (「構造的防止」と書いて一部しか防げない)
