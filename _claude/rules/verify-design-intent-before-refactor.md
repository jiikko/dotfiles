# リファクタ / 複雑度改善を提案する前に「既に意図的に選ばれた設計」でないか確認する

> **トリガー型ルール。** 「このコード複雑だな / 消したい / 分解したい / proxy 多すぎ」と
> *第一印象で構造を変えたくなった瞬間* にだけ発動する。逆にそう感じたら、手を動かす前に必ず発動させる。

## ルール

「複雑すぎる / 分解すべき / facade が肥大化」と判断する前に、提案・実装より先に以下を確認する:

1. [ ] **ドキュメント (CLAUDE.md / README / docstring)** に「意図的にこうした」と明記されていないか
2. [ ] **既存 issue / PR / 設計メモ (完了済み含む)** で同じ分解が既に検討され「やらない」と決定されていないか
3. [ ] **実 callsite** を grep して、提案が breaking change にならないか
4. [ ] **可視性 (private / internal)** を確認して、「API を変えずに論理分割」が本当に可能か
5. [ ] **行数ではなく認知負荷** (読む時の jump 数 / 追加時の touch 箇所数) で測る。大ファイルが domain 複雑度 (仕様・プロトコル・状態機械の写像) 由来なら、分解は downgrade になりうる

1 つでも「意図的」「判断済」が見つかったら、**提案を取り下げるか、その制約を踏まえて scope を絞る**。

加えて:

- **speculative refactor (先回り分解) より「実変更 trigger 待ち」を default にする**。「将来困りそうだから今分解」ではなく、trigger (「次にこの facade を触る機能追加が来た時」「N 個目の sub-component を足す時」等) を明記して実需要まで構造を凍結する
- 改善の**効能を誇張しない**。「構造的に防止できる」と書く前に全経路を防げるか確認する (一部しか防げないなら「探索範囲を局所化できる。ただし〜は手動更新が残る」と正確に書く)

## なぜ

確認を怠った第一印象の提案は空振りを量産する。典型: 「proxy が多い = 削除すべき」→ 実はドキュメントに「callsite を一度に書き換えないための意図的設計」と明記済 / 「全部抽出すべき」→ 過去 issue で「実需要で判断する」と決定済 / 「extension で論理分割できる」→ private member 依存で可視性を緩める必要があった / 大ファイル = 複雑と判定 → 中身は仕様の写像で分解すると逆に読みにくい。確認のコストは小さく、意図的設計を壊す手戻りは大きい。

このルールの効果は「やらなかったこと」に宿るため見えにくく形骸化しやすい。「複雑そう」と感じた瞬間が思い出す唯一のフック。

## 「自律改善」(CLAUDE.md) との境界

自律改善は「**触っている範囲**の明らかな劣化 (重複・到達不能・責務混在) を変更と同時に直す」。本ルールが止めるのは「**依頼と無関係に**第一印象で先回り分解する」こと。

| 状況 | どちら |
|---|---|
| 変更した関数内の到達不能コード / 自分の変更で死んだ分岐 | 自律改善 (消す) |
| 別ファイル・別レイヤーの似たコード / 「ついでにこの大きいファイルも割りたい」 | 本ルール (止まる・trigger 待ち) |

迷ったら「**今の依頼で必然的に触る範囲か?**」で切る。必然なら改善、「ついで」なら凍結。

## やること / やらないこと

- ✓ 提案前にドキュメント / 既存 issue / callsite / 可視性を確認
- ✓ speculative refactor は trigger を明記して watch に回す
- ✓ 複雑度評価 / refactor 提案は外部レビュー (codex 等) を通す
- ✗ ドキュメントに「意図的」とある構造を「複雑」と判定して分解提案
- ✗ 既存 issue で「やらない」と決定済の分解を逆転提案
- ✗ 「API 変えず論理分割」を可視性確認なしで主張
- ✗ 効能の誇張 (「構造的防止」と書いて一部しか防げない)
