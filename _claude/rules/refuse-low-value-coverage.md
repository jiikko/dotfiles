# カバレッジ向上を要求されても「テスト困難 かつ 低価値」なら拒否する

## ルール

- **「カバレッジを上げて」と要求されたとき、対象が (a) テストが著しく困難 かつ (b) テストしても有益でない、の両方を満たす場合は、カバレッジ向上を拒否する**
- 拒否は黙って skip するのではなく、**「何がテストを困難にしているか」と「なぜ低価値か」を具体的に述べ、カバレッジは目的の代理指標(proxy)であって目的そのものではない**ことを添える
- **(a)(b) のどちらか片方しか満たさないときは拒否しない**。特に「テスト困難 かつ 高価値」を「困難だから」という理由で skip する言い訳にこのルールを使ってはならない (下記マトリクス参照)

## 2 軸で判断する (テスト容易性 × 価値)

| | 高価値 (回帰で困る / バグが出やすい) | 低価値 (壊れても気づく / 自明) |
|---|---|---|
| **テスト容易** | 書く (最優先) | 書いてよいが優先度は低い |
| **テスト困難** | **テスタブルに構造を直してから書く** (拒否ではない) | **拒否する ← このルールの発動点** |

カバレッジ要求に対して、拒否してよいのは右下のセルだけ。左下 (困難×高価値) は「ロジックを pure に切り出す / seam を作る」等で**テスタブルにしてから書く**のが正解であり、困難さを理由に逃げない。

## 「テスト困難」の判断基準

- framework 内部挙動・UI・デバイス・環境に強結合していて、まともに検証できない (`no-osascript-for-ui-verification.md` / `no-ios-simulator-verification.md` と同根)
- **テストのためだけに production コードへ seam / 依存注入 / 抽象を足す必要があり、それが本番側の複雑性をむしろ上げる** (テストの都合で設計を歪める)
- 非決定性 (時刻・乱数・ネットワーク・並行) が本質で、flaky を避けられない

## 「低価値」の判断基準

- 薄い委譲 / glue コード — テストを書いても「mock が呼ばれたこと」しか assert できない (`verify-design-intent-before-refactor.md` の thin delegation と同じ判断)
- 自明な getter/setter・単なる config・定数の羅列
- 生成コード (テストすべきはジェネレータ側)
- **カバレッジの数値を上げること自体が目的化していて、回帰バグを一件も捕まえない**テスト

## 拒否の作法 (黙って skip しない)

1. どのセル (困難×低価値) に該当するかを、対象コードを指して具体的に言う
2. 「カバレッジは proxy。この行を覆っても守れる不変条件が無い」と根拠を述べる
3. 代替を出す: 「ここは薄い委譲なので、価値あるロジック X を pure 関数に切り出せばそちらは容易かつ有益にテストできる」等。単なる拒否で終わらせない

## 例外 / 誤用防止

- **回帰防止テスト (実際に起きたバグの再現) は「低価値」に含めない**。過去に発生したバグは再現条件が確定していて回帰で必ず困るため高価値。必ず書く
- ユーザーが「数値目標 (例: 80%) を必達」と明示したら従う。ただし**無益・高コストな箇所を一言明示**し、そこは薄いテストになる旨を添える (黙って水増しテストを量産しない)
- 「困難だから」を単独理由に高価値ロジックの skip に流用しない (マトリクス左下は refactor して書く)

## やること / やらないこと

- ✓ カバレッジ要求は「テスト容易性 × 価値」の 2 軸で評価し、困難×低価値のときだけ拒否する
- ✓ 拒否時は該当箇所・低価値の根拠・代替案をセットで示す
- ✓ 困難×高価値はテスタブルに構造を直してから書く
- ✗ カバレッジ数値のためだけの、回帰を捕まえない水増しテストを書く
- ✗ 「テスト困難」を高価値ロジック skip の言い訳に流用する
- ✗ 黙って skip する / 理由を述べずに拒否する
