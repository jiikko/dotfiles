# Pending / 却下した issue の理由はコード側にもコメントで残すこと

## ルール

- **issue / 監査指摘 / レビュー指摘を「対応しない」「現状維持で OK」「これは仕様」と判断した場合、その判断理由を当該コードの直近にコメントで残す**
- issue ファイル側の `❌ 該当なし` だけでは不十分。issue は移動・rename・done 退避で参照が切れるが、**コードは現場に残り続ける**。改修者が該当行を編集する瞬間に必ず目に入る場所に「なぜ素朴に直さないのか」が必要
- コメントは「結論」ではなく「**制約 (なぜ変えられないか)**」を書く。結論だけだと将来の改修者が再評価できない
- コード側に痕跡がないと、次回の自動 audit で**同じ誤指摘が再生成される** (コメントがあれば即時棄却できる)

## コメントの書き方

書くべき 3 要素:

1. **制約の発生源**を明示する。典型: 外部サービス側の登録値との一致 (OAuth Redirect URI / 課金プロダクト ID 等) / 配布・署名・審査の制約 / 別リポジトリ・別パッケージの仕様 / 同一リポジトリ内の別ファイルの不変条件
2. **どうなれば変更可能か**を一行添える (「登録値を同期更新すれば変更可能」「v2 SDK では不要になる」)
3. **依存先** (URL / ドキュメント / ファイルパス + シンボル名) を grep で追える形で書く

避けるべき書き方:

- ❌ 「変更禁止」「絶対に変えない」という断定 (将来の再評価を阻害する。「〜であるうちは変更しない。〜が変わったら再評価」と条件付きにする)
- ❌ 「過去にバグった」だけ / 「TODO: あとで調査」だけ (中身が伝わらない)
- ❌ issue 番号だけ書いて中身を書かない (issue は移動する)

依存先が**自リポジトリ内の別コード**の場合は特に慎重に: 依存先のリファクタでコメントがサイレントに形骸化しうるため、ファイルパス + シンボル名を具体的に書き、「現状の〜の実装に依存。〜変更時に再評価」と条件付きにする。

## 例 (外部登録値と一致が必要なケース)

```js
// OAuth 認証後のリダイレクトURI（ローカルサーバーで受け取る）
//
// ⚠️ ポートを変更しないこと:
// この URI は IdP 側 (App Console) に登録済みの Redirect URI と完全一致する必要があり、
// 動的割当 (port 0) や別の固定値に変えると認証フロー全体が破綻する。
// 変更が必要な場合は IdP の管理画面で Redirect URI も同じポートに更新すること。
redirectUri: getEnv('REDIRECT_URI', 'http://localhost:43823/auth/callback'),
```

何が制約か (外部登録値との一致) / 何が起こるか (OAuth エラー) / どう変えるか (外部側を同期更新) の 3 点が入っている。

## 適用タイミング

- issue / 監査指摘を「該当なし」「現状維持」でクローズした時
- レビュー指摘 (codex / cross-review) を「対応不要」と判断した時
- リファクタ提案を「現状維持」、バグ修正提案を「これは仕様」と判断した時

## やること / やらないこと

- ✓ 「なぜ素朴に変えないのか」を制約ベースで該当コード直近にコメント
- ✓ 依存先と「どうなれば変更可能か」を明示
- ✗ issue 側だけに「該当なし」を書いてコード側に痕跡を残さない
- ✗ 断定口調で将来の再評価を阻害する
