# Pending / 却下した issue の理由はコード側にもコメントで残すこと

## ルール

- **issue / 監査指摘を「対応しない」「現状維持で OK」と判断した場合**、その判断理由を **当該コードの直近にコメントで残す** こと
- issue ファイル側にも `❌ 該当なし` / `⏳ pending` の判定と理由を書くが、それだけでは不十分。issue は移動・rename・done への退避が起きうる一方、**コードは現場に残り続ける**ため、将来の改修者が「なぜ素朴に直さないのか」を読み取れる場所に痕跡が必要
- コメントは「結論」ではなく「制約 (なぜ変えられないか)」を書く。結論だけだと将来の改修者が再評価できない
- **状況は変わりうる**。外部サービス・別リポジトリ・ライブラリ仕様・別ファイルの定数などに依存している場合、依存先が変わったときに本コメントの前提も崩れる可能性がある。そのため:
  - 依存先 (外部サービスの登録値 / 別パッケージのバージョン / 別ファイルの定数 / 仕様書 など) を **明示** する
  - 「この前提が変わったらどうすべきか」を一行添える
  - 断定口調 (「絶対に変更不可」) ではなく **条件付きの留保** にする (「〜であるうちは変更しない。〜が変わった場合は再評価」)

## なぜコードにも残すか (issue だけでは不足な理由)

| 観点 | issue 側のみに残した場合 | コード側にもコメントを残した場合 |
|------|--------------------------|-------------------------------|
| 改修者が変更前に気づくか | issue を grep する習慣がない限り気づかない | 該当行を編集する瞬間に必ず目に入る |
| 監査エージェントの誤指摘再発 | 次回の自動 audit で **同じ誤指摘** が再生成される | コメントに「これは制約による必然」と書いてあるので audit 出力を即時棄却できる |
| issue ファイルの永続性 | done に退避 / 別ファイルに統合 / リネームで参照が切れる | コード自体が動く限り残る |
| 文脈の局所性 | issue を読みにいく必要 | 該当行と同じ画面で完結 |

## コメントの書き方ガイド

### 良い書き方

- 制約の **発生源** を明示する。発生源の典型カテゴリ:
  - 外部サービス側の登録値との一致が必要 (OAuth Redirect URI / 課金プロダクト ID / Webhook URL など)
  - 配布・署名・審査側の制約 (コード署名 entitlement / Notarization / ストア審査ガイドライン など)
  - 別リポジトリ / 別パッケージの仕様に依存 (依存先のバージョンや実装に紐づく)
  - 同一リポジトリ内の別ファイル・別レイヤーの不変条件に依存
- **どうなれば変更可能か** を書く (「登録値を更新すれば変更可能」「v2 SDK では不要になる」など)
- **依存先の URL / ドキュメント / 別ファイル位置** を書く

### 避けるべき書き方

- ❌ 「変更禁止」「絶対に変えない」(将来の文脈で誤判定の原因)
- ❌ 「過去にバグった」だけ (具体的に何のバグかが伝わらない)
- ❌ issue 番号だけ書いて中身を書かない (issue は移動する)
- ❌ 「TODO: あとで調査」だけ (何を調査するかが残らない)

### 状況が変わりやすいケースの慎重な書き方

依存先が **他のコード** (= 自分のリポジトリ内の別ファイル / 別パッケージ) の場合、**コメントは形骸化しやすい**ため、書き方に慎重になる必要がある。

外部サービス側の登録値や審査制約と違い、内部コードへの依存は将来別の改修者がサイレントに崩しうる (依存先のリファクタ・rename・削除が起きてもコメントは追従しない)。コメントだけが残って実態と乖離すると、むしろ「古い情報を信じて誤判断する」リスクを生む。

そのため:

- 依存先のファイルパス + 関連シンボル名を **具体的に** 残す (`path/to/foo.js` の `BAR_CONSTANT` に依存、など)。grep で追えるように書く
- 「この前提は将来変わるかもしれない」ことを示唆する (「現状の foo.js の実装に依存」「v3 への移行時に再評価」)
- 単に「変更不可」と書かず、「現状の前提下では」「〜が変わるまでは」のように条件付きにする
- コメントを書いた時点の前提が崩れていないかは、依存先を触る側 (= 別ファイルの改修時) も含めて意識する必要がある

## ルール適用タイミング

以下のいずれかが起きた時にコメントを書く:

- issue / 監査指摘を「該当なし」「現状維持で OK」とクローズした時
- レビュー指摘 (Codex review / cross-review / コードレビュー) を「対応不要」と判断した時
- リファクタリング提案を「現状維持」と判断した時
- バグ修正提案を「これは仕様」と判断した時

## やること / やらないこと

- ✓ コードに「なぜ素朴に変えないのか」を制約ベースでコメント
- ✓ 依存先 (外部サービス / 別ファイル / ドキュメント) を明示
- ✓ 「どうなれば変更可能か」を一行添える
- ✓ 他のコードに依存する制約は条件付きの留保で書く
- ✗ issue 側だけに「該当なし」を書いてコード側に痕跡を残さない
- ✗ 断定口調 (「絶対変更不可」) で将来の再評価を阻害する
- ✗ 「過去のバグ」「TODO」だけの中身のないコメント
- ✗ issue 番号だけ書いて中身を書かない (issue 退避でリンク切れる)

## 例

### 例 1: 外部サービス側の登録値と一致が必要なケース

OAuth コールバック用のローカルポートが、外部 IdP 側に登録した Redirect URI と完全一致する必要があるケース。「ポート 0 で動的割当にしては?」と提案されても対応不可。issue 側だけに「該当なし」と書いて閉じると、次回 audit で同じ提案が再生成される。

```js
// OAuth 認証後のリダイレクトURI（ローカルサーバーで受け取る）
//
// ⚠️ ポートを変更しないこと:
// この URI は IdP 側 (App Console) に登録済みの Redirect URI と完全一致する必要がある。
// IdP は登録された URI と一致しない callback を OAuth エラーとして拒否するため、
// ポートを動的割当 (port 0) や別の固定値に変更すると認証フロー全体が破綻する。
// ポート変更が必要な場合は、IdP の管理画面で該当アプリの Redirect URI も
// 同じポートに更新すること。
redirectUri: getEnv('REDIRECT_URI', 'http://localhost:43823/auth/callback'),
```

ポイント:

- **何が制約か**: 外部サービス登録値との一致
- **何が起こるか**: 一致しないと OAuth エラー
- **どう変えるか**: 外部側を同期更新すれば変更可能

### 例 2: 別ファイルの実装前提に依存するケース

```js
// 同一ファイルの並行ロードと競合しないよう、
// 現状の RequestRegistry (src/main/ipc/request-registry.js) が
// dedupeKey ベースで in-flight を join する前提で安全。
// RequestRegistry の dedupe ロジックが変わったらここも再評価が必要。
```

ポイント:

- 依存先のファイルパスとシンボルを残す
- 「現状の〜に依存」と条件付きで書き、将来の再評価ポイントを明記する
