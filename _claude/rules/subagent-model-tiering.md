# サブエージェントは sonnet が使える箇所では sonnet を使い、アウトプットは必ず main agent が検閲する

## ルール

- **サブエージェント (Agent ツール) を起動するとき、sonnet で足りるタスクには `model: "sonnet"` を指定する**。main モデル (Opus/Fable) を全タスクに使わない
- **sonnet / haiku のアウトプットは無検閲で採用しない。main agent が必ず検閲してから採用・コミットする**
- 検閲なしのコミット・報告への転記を禁止する。「エージェントが完了したので commit」は不可。diff / 成果物を main agent が読んでから

## モデル選択の目安

| タスク | モデル |
|---|---|
| 機械的な置換・定型修正・grep 系調査・ファイル列挙 | haiku / sonnet |
| 定型レビュー・ドキュメント推敲・テスト実行と結果報告 | sonnet |
| 設計判断・複雑なデバッグ・アーキテクチャ評価・微妙な判断を伴う編集 | main モデル |
| 迷ったら | sonnet で出して main が検閲 |

## 検閲の観点 (sonnet/haiku のアウトプットを採用する前に)

- **根拠のない断定の混入**: もっともらしいが検証されていない claim (「実測確認済み」「〜は存在しない」等) は裏取りする
- **参照の実在性**: エージェントが言及したファイルパス・コマンド・シンボルが実在するか
- **指示逸脱**: 担当外ファイルへの変更・指示にない創作 (新機能・新セクションの発明) が混ざっていないか
- **半端な編集**: 途中終了・部分適用で文書やコードが矛盾状態になっていないか
- 編集系タスクなら `git diff` を main agent が直接読む。報告文だけで判断しない

## なぜ

- main モデルを全サブタスクに使うとトークンコストと rate limit を無駄に消費する (session limit 到達で並行エージェントが途中死した実例: 2026-06-12)
- 一方で下位モデルは「もっともらしい誤り」の混入率が上がるため、コスト削減分は検閲で品質を担保する。分業: 下位モデルが作業し、main が判断する

## 例外

- ユーザーが明示的にモデルを指定した場合はそれに従う
- 検閲自体が再実行より高くつく極小タスク (1行の機械的修正など) は main agent が自分でやる方が速い
