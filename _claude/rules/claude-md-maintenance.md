# CLAUDE.md の保守ルール — 触ったら直す、定期レビューしない

## ルール

- **CLAUDE.md を「触ったディレクトリで読み、乖離していたら同じ PR で直す」**
- **CLAUDE.md は code の一部として扱う**。`README.md` のように後で書き直す対象ではなく、コードと一緒に更新する不変条件 (= 「ここに書かれていることは今も真である」) のドキュメント
- **定期的なレビューはしない**。「金曜に CLAUDE.md を全部見直す」のような chore 化は禁止 (ノイズが増えるだけ)。**乖離は touch base で見つけて即修正する**
- **CLAUDE.md は「Why」を保存する**。「What (= 何があるか)」はコード自体が真の出典。CLAUDE.md は **制約・過去事故・前提 (= 改修者が文脈なしでは判断を誤る部分)** を残す
- **乖離を見つけたら TODO や FIXME で残さず、その場で直す**。残すと結局放置されて信頼性が下がる

## なぜ「定期レビュー」ではなく「触ったら直す」か

| 観点 | 定期レビュー | 触ったら直す (本ルール) |
|------|------------|---------------------|
| 漏れの検出 | 全 CLAUDE.md を一度に読むと注意散漫で漏れる | 触っている範囲は文脈が新鮮なので乖離に気づきやすい |
| chore 化 | 「とりあえず日付だけ更新」のような形骸化が起きる | 必要な時にだけ動くため形骸化しない |
| 認知負荷 | 全部読み直すのに数十分〜時間 | 該当ディレクトリ 1 つだけ |
| 信頼性 | 「最後のレビューはいつ?」が分からなくなる | 「直近の commit に触れられているか」で測れる |
| Claude の効率 | 古い CLAUDE.md を信じて誤判断 | 触る前に直る = 常に新しい |

定期レビューの誘惑に駆られたら、代わりに **「直近 3 ヶ月触っていないディレクトリの CLAUDE.md を 1 件だけ読む」** ような小さな実験にとどめる。一気に全 CLAUDE.md を見直すのは禁止。

## 「乖離」とは何か (具体的な検出シグナル)

CLAUDE.md を触ったディレクトリで読んだとき、以下のいずれかが見つかったら **同じ PR で必ず修正する**。

### 構造的乖離 (= 物理的に存在しなくなったもの)

- ✗ ファイル名 / シンボル名が rename されているのに CLAUDE.md は旧名のまま
- ✗ 削除されたファイル / 関数 / プロパティが CLAUDE.md で参照されている
- ✗ ディレクトリ構成表 (「`xxx/` には A, B, C がある」) が実体と件数 / 名前で食い違う
- ✗ 「`../foo/CLAUDE.md` 参照」のリンク先が存在しない
- ✗ コードサンプル / 例として書かれた行が現コードに存在しない

### 意味的乖離 (= 物理的にはあるが、意味が変わったもの)

- ✗ 「Don't do X」と書いてあるが、X はすでに別パターンに移行済みで現コードに存在しない
- ✗ 「過去にバグった」と書かれている根拠コード (= 修正済みの workaround) が rootcause で直されてもう不要
- ✗ 「外部サービス側で〜の制約」と書かれているが、その制約はすでに緩和 / 仕様変更されている
- ✗ 「現状の foo.js の実装に依存」と書かれているが、foo.js が rewrite されて依存関係の前提が崩れた
- ✗ プラットフォーム / SDK / ライブラリのバージョンを前提にした記述が、メジャー更新で不要になった

### ぼやきポイントの陳腐化

- ✗ 「ぼやきポイント」「リファクタ候補」として書かれていた項目が、すでに解消済み

## ルール適用タイミング

以下のいずれかが起きたら **その PR / コミットの中で CLAUDE.md も同時更新する**。

| トリガー | やること |
|---------|---------|
| ディレクトリ X 配下のコードを touch (Edit / Write / 削除) | `X/CLAUDE.md` を読み、乖離していれば直す |
| ファイル rename / 移動 | rename 前のシンボル名・パスを参照している CLAUDE.md を grep して更新 |
| 関数 / クラスの削除 | 削除対象を参照している CLAUDE.md を grep して該当行を削除 or 書き換え |
| 新規ディレクトリの追加 | そのディレクトリに CLAUDE.md を新規作成するか、親の CLAUDE.md に新ディレクトリを追記するかを判断 |
| バグ修正で root cause を直した | 該当の workaround を CLAUDE.md で警告していないか確認。直っていたら警告を削除 |
| 外部サービス / SDK のバージョン更新 | バージョン依存の前提が書かれた CLAUDE.md を点検 |
| Codex / cross-review で「CLAUDE.md と実装が食い違う」と指摘された | 即時修正。指摘を rationale comment だけで閉じない |
| 「依頼と無関係だが触ったついでに気づいた」乖離 | 軽微なら直す (ぼやきポイント程度の差分なら同じ PR で OK)、大規模なら別 issue 化 |

## ディレクトリに CLAUDE.md を **作る** タイミング

新規 CLAUDE.md は **以下のいずれかを満たすディレクトリにだけ作る**。「とりあえず置く」は禁止 (空 / 形骸化 CLAUDE.md が増えるとノイズになる)。

- そのディレクトリに **そのディレクトリ固有の規約 / gotcha / 制約** が 3 個以上ある
- 親 CLAUDE.md から **頻繁に参照される** (`../foo/CLAUDE.md` 参照、と書かれる立場)
- 新しい人が読まないと **必ず事故る** 規約がある (例: 「`.background(色)` を後から重ねる時は `.allowsHitTesting(false)` 必須」)
- ディレクトリ内の認知負荷が高く (ファイル 5 個以上、責務が複数)、入口の地図が必要

逆に **作らなくていい** ディレクトリ:

- ファイル 1〜2 個でローカル規約が無い
- 親 CLAUDE.md に書ける範囲しか書くことがない
- 自動生成物 (`build/`, `tmp/`, `.build/` 等) を含むだけ

## ディレクトリの CLAUDE.md を **消す** タイミング

- そのディレクトリ自体が削除された
- そのディレクトリの規約 / gotcha が全部親に統合できる程度に縮小した
- 中身が「コードを読めば自明」だけになった (= Why が消えて What だけ残った)

## 良い書き方 / 避けるべき書き方

### 良い書き方

- 制約の **発生源** を明示する (外部サービス / 別ファイルの定数 / SDK のバージョン / 過去の incident)
- **「どうなれば変更可能か」** を一行添える (「外部側を同期更新すれば変更可能」「v3 SDK で不要になる」)
- 件数 / シンボル名 / ファイルパスを **具体的に** 書く (grep で追える形)。ただし **行番号 (`Foo.swift:42`) で位置を pin しない** — コードを 1 行足す / 消すだけで無言でドリフトし、grep-guard や「既知の合法 hit」リストが嘘になる。**file 名 + symbol 名 (or 周辺の特徴的な式)** で書き、grep / シンボル検索で追える形にする
- **既に親 CLAUDE.md にある規約を重複させない** (継承前提)
- ぼやきポイント (リファクタ候補 / 命名違和感) は **別セクション** に隔離する

### 避けるべき書き方

- ❌ 「絶対に変更不可」(将来の文脈で誤判定の原因)
- ❌ 「あとで調査する」「TODO」(中身がないと放置される)
- ❌ コードを読めば自明の What (「`xxx.swift` には `XxxView` がある」みたいな目次だけ)
- ❌ issue 番号だけ書いて中身を書かない (issue は move/rename で参照切れる)
- ❌ 「過去にバグった」だけ (具体的な再現条件 / 失敗モードを書く)
- ❌ 親 CLAUDE.md と同じ内容を再掲する (一方を更新したらもう一方の更新を忘れる)

## やること / やらないこと

- ✓ コードを touch する前 / 後に該当ディレクトリの CLAUDE.md を読む
- ✓ 乖離を見つけたら **同じ PR で** 修正する
- ✓ 修正は「結論」ではなく「制約 (なぜそうなっているか)」を書く
- ✓ 不要になった警告 (= 根本修正済みの workaround) は **削除する** (放置しない)
- ✓ ファイル rename / 削除時に `grep -r "<old-name>" **/CLAUDE.md` で参照を点検
- ✗ 定期 CLAUDE.md レビュー会を実施する (chore 化する)
- ✗ 「あとで直す」コメントを CLAUDE.md に残す (放置確定)
- ✗ 乖離を rationale comment だけで閉じる (現場が直らない = 次回も同じ指摘が出る)
- ✗ 中身が空 / 形骸化した CLAUDE.md を置きっぱなしにする (むしろ削除)
- ✗ 1 PR で複数ディレクトリの CLAUDE.md を「全部見直し」する (本来の修正と混ざってレビュー困難)
